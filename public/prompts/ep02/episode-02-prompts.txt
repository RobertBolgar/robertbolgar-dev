3. Implementation Request

[ACTUAL PROMPT]

Implement the Creator Hub video visibility fix.

IMPORTANT:

* Modify only what is necessary.
* Do not add database migrations.
* Do not delete existing records.
* Do not implement caching/backfill yet.
* Goal is production-safe display filtering.

Problem:
Creator Hubs may publicly display YouTube videos that are Unlisted because the current YouTube API calls fetch uploads playlist videos but do not request or filter by `status.privacyStatus`.

Required fix:

1. In [functions/api/public-links-by-subdomain.js](cci:7://file:///c:/Users/rober/OneDrive/Desktop/tubelinkr/functions/api/public-links-by-subdomain.js:0:0-0:0), update all `videos.list` calls used for public Creator Hub display to request:

`part=snippet,statistics,status` 

or, where statistics are not needed:

`part=snippet,status` 

2. For the Featured Video:

* Fetch the selected `featured_video_id` with `status`.
* Only display it if:

`item.status?.privacyStatus === "public"` 

* If the featured video is missing, private, unlisted, deleted, or has unknown privacy status, do not display it.
* Fall back to auto-selecting from public videos if current logic supports fallback.

3. For More Videos:

* After fetching video details with `videos.list`, filter results server-side.
* Only include videos where:

`item.status?.privacyStatus === "public"` 

* Exclude all videos where privacy status is:

  * `unlisted` 
  * `private` 
  * missing
  * unknown

4. Do not expose `privacy_status` publicly unless needed for debugging.
   The public API response should simply omit non-public videos.

5. Add defensive handling:

* If YouTube does not return `status`, treat the video as non-public.
* If all videos are filtered out, return an empty videos array.
* Do not throw user-facing errors just because no public videos are available.

6. Check for duplicate Creator Hub video-fetch logic in this same file and apply the same public-only filtering anywhere videos are returned to the public hub.

7. Testing:

* Confirm public videos still appear.
* Confirm unlisted videos no longer appear.
* Confirm selected featured unlisted video does not appear.
* Confirm More Videos does not include unlisted/private videos.
* Confirm the Creator Hub page does not break if no public videos are available.

Deliver:

* Files changed
* Exact logic added
* Confirmation that only PUBLIC videos appear on public Creator Hubs
* Any remaining risk or follow-up recommendations

8. Username Gating Fix Implementation

[ACTUAL PROMPT]

Implement a low-risk username gating fix for TubeLinkr.

Context:
Read-only audit showed usernames are required internally and legacy routes still depend on them. Existing user count is small:
- founder confirmed username: 2
- founder auto-generated username: 1
- free auto-generated username: 2
- free confirmed username: 1

Do not migrate existing users.
Do not delete usernames.
Do not make username nullable.
Do not change legacy routing.
Do not break Smart Links, Proofs, Creator Hubs, or worker routing.

Goal:
Prevent future free users from claiming/changing custom usernames while preserving existing usernames.

Requirements:

1. Backend enforcement
- In the username update endpoint, block username changes unless the user has Pro, Pro Plus, Founder, or Admin access.
- Use authoritative backend plan/founder/admin status.
- Do not trust frontend plan state.
- If a free user attempts to update username directly through the API, return a clear 403 response.
- Existing free users may keep their current username, but cannot change it.

2. Username availability endpoint
- Prevent free users from checking/claiming usernames if that endpoint is tied to username setup.
- If needed, return a response that says custom usernames require Pro.
- Do not expose username availability as a free reservation mechanism.

3. Signup/sync flow
- Continue auto-generating usernames if the database requires username NOT NULL.
- Auto-generated usernames should remain internal for free users.
- Do not set username_confirmed_by_user = 1 for free users during signup.
- Do not let free users confirm usernames during setup.

4. Frontend
- Display Name remains editable for all users.
- Username field should be shown as Pro-only or disabled for free users.
- Add copy:
  "Custom usernames are available on Pro."
- Pro/Founder users can edit username normally.
- Free users should not be forced through username setup if they cannot claim one.

5. Existing users
- No migration.
- No cleanup.
- Do not clear the one existing free confirmed username.
- Preserve existing data.

6. Tests
